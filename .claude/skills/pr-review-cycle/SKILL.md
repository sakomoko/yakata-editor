# PR Review Cycle スキル

GitHub ActionsのClaude Code Reviewが投稿するレビューコメントを自動的に待機・取得し、指摘があれば修正してコミット・プッシュする。

## トリガー条件

以下の状況で**プロアクティブに**このスキルを呼び出す（ユーザーの明示的な指示を待たない）：

- `gh pr create` を実行した直後
- `git push` を実行した直後（PRが既に存在する場合）
- ユーザーが「PRを作って」「プッシュして」と依頼し、完了した直後

## レビュー待機・取得フロー

### 1. PR番号の特定

```bash
gh pr view --json number -q '.number'
```

PR番号が取得できない場合（PRが存在しない場合）はスキルを終了する。

### 2. 現在のコメント数を記録

レビュー待機前に既存のコメント数を記録し、新しいコメントのみを検出する。

```bash
gh api repos/{owner}/{repo}/issues/{pr_number}/comments --jq 'length'
```

### 3. レビュー待機ループ

3分間隔で最大3回（合計最大約9分）待機し、新しいコメントを確認する。

```bash
sleep 180
gh api repos/{owner}/{repo}/issues/{pr_number}/comments --jq 'length'
```

- 新しいコメントが増えていたら → ステップ4へ進む
- 増えていなければ → 再度sleepしてリトライ（最大3回）
- 3回リトライしてもコメントがない場合 → 「レビューコメントが確認できませんでした」と報告して終了

### 4. レビューコメントの取得

新しいコメントのみを取得する。

```bash
gh api repos/{owner}/{repo}/issues/{pr_number}/comments --jq '.[-N:][] | {user: .user.login, body: .body}'
```

Nは新しいコメント数（現在のコメント数 - 記録時のコメント数）。

### 5. レビュー内容の分析

取得したコメントを分析し、以下を判断する：

- **修正が必要な指摘**: コード品質、バグ、セキュリティ等の具体的な改善提案
- **情報提供のみ**: 称賛やFYI的なコメント（修正不要）

修正が必要な指摘がない場合は「レビューで問題は指摘されませんでした」と報告して終了。

### 6. 自動修正

指摘内容に基づいてコードを修正する。修正時は以下を守る：

- CLAUDE.mdのルール（Strict TypeScript、コードスタイル等）に従う
- 修正後に `npm run typecheck` と `npm test` を実行して品質を確認
- 修正が妥当かどうか判断し、明らかに不適切な指摘はスキップしてユーザーに報告する

### 7. 修正のコミット・プッシュ

```bash
git add <修正したファイル>
git commit -m "fix: レビュー指摘の修正 - <修正内容の要約>"
git push
```

### 8. 修正後の再レビュー待機

プッシュすると再度GitHub Actionsのレビューが走るため、ステップ2に戻って同じフローを繰り返す。

**ただし、修正サイクルは最大3回まで**とする。3回修正してもまだ指摘がある場合は、残りの指摘内容をユーザーに報告して終了する。

## 注意事項

- `sleep` コマンドでの待機中はユーザーに「レビュー待機中...」と伝える
- 修正サイクル全体（待機+修正+再待機）が長くなりすぎないよう、修正は最大3サイクル
- レビューコメントの投稿者がbotかどうかは問わず、新しいコメントすべてを確認する
- 修正の判断に迷う場合はユーザーに確認する
